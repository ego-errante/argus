# The Run: a single-session orchestrator, Run-ID-prefix payload keying, and sent injected failures

The Run (Day 11) is one recording session that produces the graded Lifecycle Log of ‚â•10 real bundle Submissions including the required ‚â•2 failures (CONTEXT.md). A single orchestrator drives a scripted sequence under one **Run ID** ‚Äî the 3 deterministic injections (`expired_blockhash`, `compute_exceeded`, `bundle_failure`) plus `ARGUS_RUN_CLEAN_COUNT` clean Payloads (default 7) ‚Äî calling the already-proven `submit_and_track_one` / `injection_run` units **serially, each Submission tracked to finalized**, then auto-exports the Lifecycle Log (JSONL + Markdown). Three decisions here are load-bearing and non-obvious: how Payloads are keyed within one Run, that the injected failures are now **sent on the wire**, and that the orchestrator reuses the tracking path untouched.

## Why

The graded artifact must be a *coherent, reproducible* set of ‚â•10 Submissions, not "whatever rows happen to be in the DB." That pushes to a single orchestrator emitting the whole Run from one command, which forces two sub-problems the per-invocation lifecycle path never had:

- **Many Payloads under one Run, but the schema keys decisions on `(run_id, attempt)` with no Payload discriminator.** Each injection records its decision at `attempt = 1`; three injections under one literal `run_id` collide at `(run_id, 1)`, and the export can't tie a Remedy to its faulted Submission. The submission nonce (`argus-{run_id}-jito-{attempt}`) collides the same way across clean Payloads.
- **The injected failures were sim-only (ADR 0010), which contradicts the glossary and undercuts the count.** A faulted attempt-1 was preflight-simulated for classification and recorded as a `submissions` row but **never sent to the Block Engine** ‚Äî yet CONTEXT.md defines a Submission as "one bundle *sent* to the Jito Block Engine." Worse, if sim-only rows don't count toward "‚â•10 *real* bundle submissions," then 3 injections + 7 clean yields only **9** actually-sent bundles (7 clean + 2 recoveries; `bundle_failure` aborts with no attempt-2) ‚Äî under the bar.

The chosen resolutions keep the proven submit/track/persist path (ADR 0004/0009/0010) completely unchanged; only the orchestration loop, one `injection_run` send, and the export are new.

## Considered options

- **Payload keying ‚Äî single literal `run_id` + a Payload discriminator vs. Run-ID prefix.** Adopted the **prefix**: the Run is identified by a Run ID (`run-{ts}`) and Payload *k* runs under a derived child `run_id = run-{ts}-p{k}`; within a Payload, `attempt` 1/2 and the existing nonce are unchanged. This makes `(run_id, attempt)` unique per Payload (no decisions collision) and the nonce unique per Submission **with zero schema change and zero edits to `submit_and_track_one`/`injection_run`/`record_decision`** ‚Äî the orchestrator just varies the `run_id` it passes; the export selects `WHERE run_id LIKE 'run-{ts}-%'`. Rejected the single-literal-`run_id`-plus-`payload_seq`-column: a marginally tidier data model bought with a schema migration *and* surgery on the exact tracking unit we deliberately reuse. The prefix bends "one run_id per Run" into "one Run *prefix*," accepted as a legible Run ‚Üí Payload ‚Üí Attempt id hierarchy. It also cleanly excludes the scattered rows from earlier ad-hoc runs (a fresh prefix).
- **Injected failures ‚Äî sim-only vs. sent.** Adopted **sent**: the faulted attempt-1 goes on the wire (send-and-record, *not* tracked-to-finalize ‚Äî a doomed bundle never emits a Landed event, so a finalize stream would just burn ~30 s waiting), so each injected failure is a real, non-landing Submission that counts toward ‚â•10 and is genuine ‚â•2-failure evidence. **This amends ADR 0010's sim-only stance.** The preflight simulation stays exactly where it is *for classification* ‚Äî ADR 0010's core reason holds (an all-or-nothing bundle leaves no on-chain meta to read), we simply stop *withholding* the doomed bundle from the Block Engine. It is free: tips are paid only on landing, and a tx that never lands is never charged. The send is **scoped to the Run** (a `send_faulted` flag the orchestrator passes; the standalone `ARGUS_INJECT` path keeps ADR 0010's sim-only behavior). Non-landing is authoritative from the **absent `landed_slot`** on the persisted row (the export renders it as `‚Äî`); an inflight `getInflightBundleStatuses` cross-check was considered but dropped ‚Äî fired right after submit it reads before Jito indexes the bundle, so it adds per-region round-trips for no firmer evidence than the row already carries.
- **Tracking ‚Äî serial-to-finalized vs. pipelined.** Adopted **serial**, reusing `submit_and_track_one` verbatim: the per-Submission ~30 s finalize wait naturally spaces the Jito rate limit and leader-window alignment, so no new pacing logic, and the showcase processed‚Üíconfirmed‚Üífinalized deltas come out complete. A Run of ~12 is several minutes wall-clock ‚Äî acceptable for a one-time recording session. Pipelining (one stream watching many signatures) was rejected: a real rewrite of the proven path whose only benefit is speed a single Run doesn't need.
- **Export trigger ‚Äî auto-only vs. standalone-only vs. both.** Adopted **both**: the orchestrator auto-exports at end-of-Run (one command ‚Üí Run *and* Log), and a standalone `ARGUS_EXPORT=run-{ts}` regenerates the Log from the persisted DB without re-running. A Run costs real SOL; the re-export lets the Markdown/JSONL formatting be iterated for free from SQLite (the source of truth, ADR 0004).

## Consequences

The orchestrator is a thin serial loop: for each Payload it mints `run_id = run-{ts}-p{k}` and calls the existing unit, so `submissions`/`decisions` keep their current keys and the tracking/persistence code is untouched. **Run identity is a prefix, not a column value** ‚Äî every query that scopes "a Run" must filter `run_id LIKE 'run-{ts}-%'`, and the Run ID is the `-p{k}`-stripped prefix; this is the one place the Run ‚Üí Payload ‚Üí Attempt hierarchy is implicit in a string rather than explicit in schema. The Lifecycle Log is a two-part artifact: a flat Submissions table (slot ‚Üí Solscan block, signature ‚Üí Solscan tx, `‚Äî` for non-landed rows) plus a separate Agent-Decisions section carrying each Failure's Remedy, confidence, model, rationale, and a Reasoning-Trace excerpt (the full trace lives in the JSONL). Because injected failures are now sent, the table's non-landed rows are real on-the-wire Submissions, not preflight artifacts. Robustness is deliberately minimal but fail-fast where it matters: the orchestrator pings the Agent `/health` and refuses to start if it's down (a scored Run must not silently degrade to `local-fallback`, ADR 0006), warns on a thin payer balance, then runs best-effort ‚Äî an organic non-landing of a clean Payload is recorded and the Run continues (no per-payload retry; the default 2-Submission margin over 10 absorbs it), a stream stall self-terminates via the ADR 0009 ceiling, and only fatal setup errors (keypair, DB) hard-stop. At the end the orchestrator asserts ‚â•10 sent and ‚â•2 failures and warns loudly if either is short, so a deficient Run is caught before submission rather than at Lifecycle-Log assembly.
